# Deployment

From an empty GitHub repo to a live preview URL. Nothing here is undocumented, and nothing needs a decision you have not already made.

---

## 0. Before you start

You need three things. The first is required; the other two can wait.

| | |
| --- | --- |
| A Supabase project | Required for orders to save. |
| A WhatsApp Business number | Required for the hand-off. The order still saves without it. |
| A Sentry DSN / GA4 id | Optional. Leave empty and neither is loaded. |

---

## 1. Push to GitHub

```bash
cd turjoman
git init
git add .
git commit -m "Turjoman V1"
git branch -M main
git remote add origin git@github.com:<you>/turjoman.git
git push -u origin main
```

`.gitignore` already excludes `node_modules`, `.next`, `.vercel`, and **every** `.env` file. `.env.example` is committed — it is the contract; the values are yours.

---

## 2. Prepare Supabase

In the SQL editor, run both migrations **in order**:

1. `supabase/migrations/20260713000000_orders.sql` — the `orders` table, its check constraints, its indexes, RLS on with no policies.
2. `supabase/migrations/20260713010000_order_reference.sql` — the `TRJ-XXXXXXXX` reference.

Then take two values from **Project Settings → API**:

- **Project URL** → `SUPABASE_URL`
- **`service_role` key** → `SUPABASE_SERVICE_ROLE_KEY`

> The `service_role` key bypasses RLS. It belongs on the server and nowhere else. It is never prefixed `NEXT_PUBLIC_`, so it cannot enter the browser bundle — and the write endpoint re-validates every payload before that key touches the database.

You do **not** need the `anon` key. The browser never talks to Supabase.

---

## 3. Import into Vercel

**New Project → Import Git Repository → select the repo.**

Vercel detects Next.js and fills these in. Accept them:

| | |
| --- | --- |
| Framework preset | **Next.js** |
| Install command | `npm install` |
| Build command | `npm run build` *(i.e. `next build`)* |
| Output directory | `.next` — **leave it alone**, Vercel handles this |
| Node version | 20.x or 22.x |
| Root directory | `./` |

**No `vercel.json` is needed and none is included.** Everything it would configure is already in the code: security headers in `next.config.ts`, the CSP in `middleware.ts`, caching on the route handlers. A `vercel.json` here would be a second, quieter place for the same truth.

---

## 4. Environment variables in Vercel

**Settings → Environment Variables.** Add to **Production** and **Preview** both.

### Required

| Name | Value | Scope |
| --- | --- | --- |
| `SUPABASE_URL` | `https://<project>.supabase.co` | Server |
| `SUPABASE_SERVICE_ROLE_KEY` | the `service_role` key | **Secret** |
| `WHATSAPP_BUSINESS_NUMBER` | `966546040520` | Server |

### Optional — off unless set

| Name | Effect when empty |
| --- | --- |
| `NEXT_PUBLIC_GA4_MEASUREMENT_ID` | No analytics script, no beacon, no cookie |
| `NEXT_PUBLIC_GTM_ID` | Same. Wins over GA4 if both are set |
| `NEXT_PUBLIC_SENTRY_DSN` | Monitoring is a no-op |
| `SENTRY_DSN` | Same, for the server. Falls back to the public one |

### Set automatically — do not add these

`NEXT_PUBLIC_RELEASE`, `NEXT_PUBLIC_BUILD_ID`, `NEXT_PUBLIC_BUILD_TIME`, `NEXT_PUBLIC_COMMIT_SHA` are derived at build time in `next.config.ts` from Vercel's `VERCEL_GIT_COMMIT_SHA`. Setting them by hand only breaks the link between a crash report and the deploy that caused it.

### Which of these is actually a secret

Exactly one: **`SUPABASE_SERVICE_ROLE_KEY`**. Mark it as a secret in Vercel.

The WhatsApp number is on your storefront. The Sentry DSN can only write, never read. The GA4 id is in the page source of every site that uses it. They are configuration, not secrets — but they still belong in the environment, because they change and the code should not.

---

## 5. Deploy

Push to `main` → production. Open a pull request → **Vercel comments a Preview URL on the PR**. That preview is a real deploy against the same code, with the Preview environment variables.

---

## 6. Verify the deploy

| Check | Expected |
| --- | --- |
| The flow runs end to end | Hero → Order, no page loads |
| Submit a real order | A row lands in `orders` with `status = new` |
| The reference | `TRJ-XXXXXXXX`, generated by the database |
| WhatsApp | Opens pre-filled, addressed to your number |
| `curl -I <url>` | `content-security-policy` with a `nonce-…`, `strict-transport-security`, `x-frame-options: DENY` |
| Rate limit | The 6th order POST in a minute returns `429` |

If an order fails, the founder sees a typed Arabic message under the submit button — not a stack trace, and not silence.

## 7. Source maps (only if you use Sentry)

`productionBrowserSourceMaps: true` is already set, so the maps are built. Upload them on deploy, tagged with the same release the app reports:

```bash
npx @sentry/cli sourcemaps inject .next/static/chunks
npx @sentry/cli sourcemaps upload \
  --release "$VERCEL_GIT_COMMIT_SHA" \
  --dist "$(cat .next/BUILD_ID)" \
  .next/static/chunks
```

Without this a stack trace is a wall of minified letters. With it, frames resolve back to the original TypeScript.

---

## Rollback

Vercel keeps every deploy. **Deployments → the last good one → Promote to Production.** The database is unaffected: orders are append-only and no migration in this release drops anything.

---

## One build-time dependency worth knowing about

`app/layout.tsx` uses `next/font/google`. Next **downloads** Almarai, IBM Plex Sans Arabic and Cormorant Garamond at build time and self-hosts them into the bundle — so the running site never calls Google, but **the build machine must be able to reach `fonts.googleapis.com`.**

Vercel and GitHub Actions can. An air-gapped CI cannot, and `next build` will fail there with:

```
`next/font` error: Failed to fetch `Almarai` from Google Fonts.
```

If you ever need an offline build, the fix is to vendor the three `.woff2` files into `public/fonts/` and switch `layout.tsx` to `next/font/local`. That is a change to a frozen file, so it needs your approval — it is not in this release.

## What is *not* automated

Three things, deliberately:

1. **The migrations.** They are run by a human against a database, once. An app that migrates its own production schema on boot is an app that can destroy it on boot.
2. **The source-map upload.** It needs a Sentry auth token; add it to CI when you add Sentry.
3. **The hero plate.** `public/hero/` ships pre-built. Regenerate it only if the photograph changes: `npm run plate -- ./assets/atelier-master.png`, then paste the printed LQIP into `lib/hero-plate.ts`. The master render is not in git — it is 3000px of nothing the app needs at runtime.
